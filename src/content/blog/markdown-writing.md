作为入门开发者，我现在最需要的是把知识整理清楚，而不是追求复杂工具。

Markdown 很适合这个阶段：语法少、文件轻、能直接放进博客，也方便以后迁移。

## 一篇技术笔记的结构

我准备固定使用这个格式：

```md
---
title: '文章标题'
description: '一句话说明这篇文章解决什么问题'
pubDate: '2026-06-25'
category: '学习笔记'
tags: ['Markdown', 'Astro']
---

## 问题

这里写我遇到的问题。

## 过程

这里写我怎么查、怎么试。

## 结果

这里写最后怎么解决。

## 下次注意

这里写复盘。
```

## 为什么这样写

很多学习笔记失败在太散。今天记一个命令，明天贴一段代码，过一周自己也看不懂。

固定结构以后，每篇文章都能回答四个问题：

1. 我为什么写这篇
2. 我做了哪些尝试
3. 最后结果是什么
4. 下次能不能少走弯路

## 代码块很重要

技术笔记最好保留能复现的命令。

```bash
npm run build
npm test
```

只写“构建成功了”没有用，写下具体命令，以后才能知道当时怎么验证的。

## 我的规则

- 一篇文章只解决一个问题
- 标题写具体，不写“学习记录 1”
- 保留错误信息和最终命令
- 文章最后写一句复盘

这套规则不复杂，但能让博客更像开发记录，而不是散乱日记。
