学开发时，真正容易忘的不是大概念，而是细节：命令怎么写、配置放哪里、报错是什么意思。

所以我准备给自己固定一个学习笔记模板。

## 模板

```md
# 标题：具体到一个问题

## 背景

我为什么遇到这个问题？

## 环境

- 系统：
- Node 版本：
- 项目路径：
- 相关工具：

## 操作步骤

1. 第一步
2. 第二步
3. 第三步

## 报错或现象

把关键报错原样贴出来。

## 解决办法

写最终有效的方法。

## 复盘

下次遇到类似问题先检查什么？
```

## 为什么要写环境

同一个命令，在不同系统和不同版本里可能表现不一样。如果只写“运行失败”，以后很难判断原因。

比如 GitHub Pages 的部署问题，就和项目路径、仓库名、Astro 的 `base` 配置有关。

## 为什么要保留报错

报错信息看起来吓人，但它是定位问题最直接的线索。以后搜索时，原始报错比自己的描述更准确。

## 适合记录什么

- VS Code 插件配置
- Git 和 GitHub 命令
- Chrome 调试过程
- Astro 构建报错
- GitHub Pages 发布问题

## 复盘标准

一篇笔记写完后，我会问自己一句：

> 如果一个月后再遇到这个问题，我能不能只看这篇文章就解决？

如果答案是否定的，就说明这篇笔记还不够清楚。
