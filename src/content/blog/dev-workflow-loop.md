继续搭建博客时，我不想只追求页面看起来更复杂，而是希望每次改动都能形成一个稳定流程。

这篇记录整理的是我现在使用的开发闭环：先判断要解决什么，再做小范围改动，然后本地构建和测试，最后发布到线上检查真实页面。

## 这次目标

博客已经有首页、文章页、分类页和关于页，但如果继续长期维护，只靠“看起来不错”还不够。

我给自己定了三个目标：

- 页面风格要保持统一
- 文章阅读体验要更完整
- 每次发布前都要有可执行验证

这三个目标能帮助我避免一个常见问题：功能越加越多，但不知道哪里真正可用。

## 我的操作顺序

我现在会按这个顺序开发：

1. 先读项目规则和现有代码
2. 找出最影响体验的一处问题
3. 只改和目标直接相关的文件
4. 给关键行为补测试
5. 构建成功后再发布
6. 打开线上地址做最终检查

这个流程不复杂，但对入门开发者很重要。它能让我知道每一步为什么做，而不是凭感觉到处改。

## 为什么要先做小改动

博客是一个长期项目，最怕一次改太多。改动太大时，页面出了问题就很难判断是哪一处导致的。

所以我更愿意把任务拆小。例如这次先完善文章详情页，下一次再考虑评论、搜索、站点地图说明或个人项目页。

小改动的好处是：

- 更容易测试
- 更容易回退
- 更容易写清楚复盘
- 更容易保持页面风格一致

## 验证标准

每次发布前，我至少会检查两件事：

```bash
npm run build
npm test
```

`npm run build` 用来确认 Astro 能生成静态页面，`npm test` 用来检查关键页面里是否还保留了预期结构和内容。

发布之后，还要打开真实线上地址。因为本地构建成功不代表 GitHub Pages 的路径、图片和链接都正常。

## 这次学到什么

我开始理解，开发不是“写完代码就结束”，而是一个闭环：

```text
需求判断 -> 小范围实现 -> 自动检查 -> 发布 -> 线上验证 -> 复盘记录
```

当这个闭环稳定以后，博客就不只是一个页面，而是我练习开发流程的地方。
